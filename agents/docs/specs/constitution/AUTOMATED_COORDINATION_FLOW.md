# 银河导航员自动化协同流程

**文档版本**: V3.16.0  
**创建日期**: 2026-03-26  
**说明**: 定义银河导航员自动调度和协同其他 8 个智能体的完整流程

---

## 🎯 核心机制

银河导航员作为 Hub，通过以下机制自动化协同其他智能体：

1. **自动触发**: 收到用户消息后自动判断需要召唤的智能体
2. **顺序调度**: 按工作流顺序依次召唤智能体
3. **状态追踪**: 通过 `.tasks/index.md` 追踪所有任务状态
4. **异常处理**: 检测失败/超时任务并自动处理

---

## 🔄 自动化协同流程

### 阶段 1: 需求接收与判断

```
用户消息
    │
    ▼
┌─────────────────────────────────┐
│ 银河导航员接收消息               │
│ 1. 解析用户意图                  │
│ 2. 判断需求类型                  │
│    - 新需求 → 标准构建流         │
│    - 简单任务 → 快速执行流       │
│    - 查询/咨询 → 直接响应        │
└──────────────┬──────────────────┘
               │
               ▼
    分配需求 ID（REQ-{序号}）
```

### 阶段 2: 自动召唤智能体（标准构建流）

```
┌─────────────────────────────────┐
│ Step 1: 召唤需求澄清 🎯          │
│ - sessions_spawn(requirement-clarification)
│ - 任务：产出《澄清提案》         │
│ - 状态：⏳ pending → 🔄 running  │
└──────────────┬──────────────────┘
               │
               ▼
    等待完成 → 更新状态 ✅ completed
               │
               ▼
    用户确认提案（暂停等待）
               │
               ▼
┌─────────────────────────────────┐
│ Step 2: 召唤需求理解 💡          │
│ - sessions_spawn(requirement-understanding)
│ - 任务：产出 OpenSpec 规约        │
│ - 状态：⏳ pending → 🔄 running  │
└──────────────┬──────────────────┘
               │
               ▼
    等待完成 → 更新状态 ✅ completed
               │
               ▼
    用户确认规约（暂停等待）
               │
               ▼
┌─────────────────────────────────┐
│ Step 3: 召唤需求解决 🪄          │
│ - sessions_spawn(requirement-resolution)
│ - 任务：按 tasks.md 执行开发     │
│ - 状态：⏳ pending → 🔄 running  │
└──────────────┬──────────────────┘
               │
               ▼
    等待完成 → 更新状态 ✅ completed
               │
               ▼
┌─────────────────────────────────┐
│ Step 4: 召唤需求验收 🔍          │
│ - sessions_spawn(requirement-acceptance)
│ - 任务：按 AC 逐项验收            │
│ - 状态：⏳ pending → 🔄 running  │
└──────────────┬──────────────────┘
               │
               ▼
    等待完成 → 更新状态 ✅ completed
               │
               ▼
    用户确认验收（或 override）
               │
               ▼
┌─────────────────────────────────┐
│ Step 5: 召唤需求交付 📦          │
│ - sessions_spawn(requirement-delivery)
│ - 任务：Git 提交 + 部署           │
│ - 状态：⏳ pending → 🔄 running  │
└──────────────┬──────────────────┘
               │
               ▼
    等待完成 → 更新状态 ✅ completed
               │
               ▼
    银河导航员整合汇报 → 用户
```

### 阶段 3: 进度跟踪（V3.16.0 新增）

```
┌─────────────────────────────────┐
│ 后台心跳检查（每 3-5 分钟）        │
│ 1. 读取 .tasks/index.md          │
│ 2. 检查所有 🔄 running 状态任务   │
│ 3. 计算超时：now() - last_update │
│ 4. 超时判断：                     │
│    - <30 分钟 → 继续等待         │
│    - ≥30 分钟 → 标记停滞         │
└──────────────┬──────────────────┘
               │
               ▼
    发现停滞任务
               │
               ▼
┌─────────────────────────────────┐
│ 处理停滞任务                     │
│ 1. 发送确认消息给子会话          │
│ 2. 等待响应（5 分钟）             │
│ 3. 无响应 → 标记 ❌ failed       │
│ 4. 通知用户 + 审计智能体         │
└─────────────────────────────────┘
```

## 📬 任务分发协议

### sessions_spawn 调用规范

```python
# 通用模板
sessions_spawn(
    runtime="acp" | "subagent",     # 根据任务复杂度选择
    agentId="<智能体 ID>",
    label="<智能体>-REQ-{ID}",
    task="<具体任务描述>",
    mode="run"                       # 或 "session"
)
```

### sessions_send 调用规范

```python
# 向已创建的子会话发送指令
sessions_send(
    sessionId="<session-id>",
    message="<任务指令>",
    attachments=["<文件路径>"]      # 可选：附加规约文件
)
```

## 📊 状态管理

### 任务状态流转

```
⏳ pending
    │
    │ sessions_spawn 创建会话
    ▼
🔄 running
    │
    │ 智能体执行任务
    ▼
┌───────────┬───────────┬───────────┬───────────┐
│ ✅        │ ❌        │ 🚫        │ 🚫        │
│ completed │ failed    │ blocked   │ cancelled │
└───────────┴───────────┴───────────┴───────────┘
```

### 状态更新时机

| 状态 | 更新时机 | 更新方 |
|------|---------|--------|
| ⏳ pending | 创建任务文件时 | 银河导航员 |
| 🔄 running | sessions_spawn 完成后 | 银河导航员 |
| ✅ completed | 子会话回报完成时 | 子智能体 |
| ❌ failed | 任务失败/超时时 | 银河导航员 |
| 🚫 blocked | 遇到阻塞时 | 子智能体 |
| 🚫 cancelled | 用户取消时 | 银河导航员 |

## 🚨 异常处理机制

### 任务失败处理

```
任务失败（❌ failed）
    │
    ▼
┌─────────────────────────────────┐
│ 银河导航员决策                   │
│ 1. 分析失败原因                  │
│ 2. 选择处理策略：                │
│    - 可重试 → 创建新任务重试     │
│    - 需调整 → 召唤需求澄清       │
│    - 不可恢复 → 上报用户         │
└─────────────────────────────────┘
```

### 任务超时处理

```
任务超时（>30 分钟无更新）
    │
    ▼
1. 发送确认消息 → 子会话
    │
    │ 5 分钟内响应
    ├─→ 继续执行
    │
    │ 无响应
    ▼
2. 标记 ❌ failed
    │
    ▼
3. 通知审计智能体 → 记录异常
    │
    ▼
4. 通知用户 → 等待决策
```

### 用户取消处理

```
用户取消任务
    │
    ▼
1. 更新状态 → 🚫 cancelled
    │
    ▼
2. sessions_send → 通知子会话停止
    │
    ▼
3. 清理临时文件
    │
    ▼
4. 记录取消原因
```

## 📋 并行任务管理

### 并发控制

| 限制项 | 值 | 说明 |
|--------|-----|------|
| 单智能体并发上限 | 5 个任务 | 超出后新任务排队 |
| 全局并发上限 | 20 个任务 | 超出后新任务 pending |
| 优先级 | P0 > P1 > P2 | P0 任务优先调度 |

### 多需求并行

```
时间线：
09:00  REQ-A 需求澄清 → sessions_spawn → 子会话 1

09:10  REQ-A 需求理解 → sessions_spawn → 子会话 3
09:15  REQ-B 需求理解 → sessions_spawn → 子会话 4
       ...
       
.tasks/index.md:
| 任务 ID | 智能体 | 状态 | 需求 ID |
|--------|--------|------|---------|
| task-001 | clarification | ✅ | REQ-001 |
| task-002 | understanding | 🔄 | REQ-001 |
| task-003 | clarification | ✅ | REQ-002 |
```

## 🔍 审计监督

### 自动审计触发

| 触发条件 | 审计类型 | 说明 |
|---------|---------|------|
| 任务失败 | 事后审计 | 分析失败根因 |
| 任务超时 | 实时审计 | 检测异常行为 |
| 用户取消 | 事后审计 | 分析取消原因 |
| 周期触发 | 定期审计 | 每 2 小时检查 |

### 审计报告

审计智能体定期产出《审计报告》，包含：
- 任务成功率统计
- 超时任务分析
- 违规行为检测（无规约开发等）
- 改进建议

## 📈 协同效率指标

| 指标 | 计算方式 | 目标值 |
|------|---------|--------|
| 任务成功率 | completed / total | >90% |
| 平均完成时间 | Σ(completed_time - start_time) / count | 依任务类型 |
| 超时率 | timeout_count / total | <5% |
| 用户确认停顿次数 | 确认节点暂停次数 | 依流程 |
