## 🎯 变更背景与问题

### 问题 1：任务感知不到

大总管指派 V3.14.0 宪法升级任务给「事后诸葛亮」时，只在主智能体会话中提及，而目标智能体没有感知到。

**根因**：大总管的任務安排方式不统一，导致任务传递不可靠。

### 问题 2：任务状态不透明

- 任务丢失：大总管发了一个任务，不知道智能体收到没
- 重复处理：同一个任务可能被处理两次
- 状态不明：任务在处理中？完成了？失败了？不知道

### 问题 3：任务与需求关联缺失

一个需求由多个智能体协同完成，但任务之间缺乏需求维度的关联，导致上下文不连续。

---

## ✅ 解决方案：Hub-Spoke + Master-Worker 混合模式

### 核心架构

```
大总管（Hub / Coordinator）
    ├── 消息分发 → 需求澄清智能体
    │               ├── Subagent 1 → REQ-001 需求澄清
    │               └── Subagent 2 → REQ-002 需求澄清
    ├── 消息分发 → 需求理解智能体
    │               └── Subagent 3 → REQ-001 需求理解
    └── 消息分发 → 总结反思智能体
                    └── Subagent 4 → 复盘报告
```

### 协同方式：文本 + 共享文件

```
1. 大总管接收需求（文本）
      ↓
2. 发送给 目标智能体（via sessions_send）
      ↓
3. 目标智能体 spawn subagent，并行处理
      ↓
4. Subagent 产出文件到 story/ 目录
      ↓
5. 智能体回报大总管：
   「✅ 完成了 xxx，📄 产出 story/REQ-001-blueprint.md」
      ↓
6. 大总管把「文件路径」发给下一步的智能体
      ↓
7. 继续流转...
```

## 🔗 与 Story File 的关系（互补设计）

Hub-Spoke Task Management 与 Story File 是**互补关系**，解决不同维度的问题：

| 维度 | Story File (story/state.md) | Hub-Spoke Task Management (.tasks/) |
|------|-----------------------------|-----------------------------------|
| **定位** | 需求开发过程的上下文容器 | 任务分配与状态跟踪 |
| **视角** | **需求视角**（一个需求的完整生命周期） | **任务视角**（谁在做什么任务） |
| **核心关注** | 需求进展到哪个阶段、上下文如何传递 | 任务状态、有没有阻塞、谁在做 |
| **粒度** | 按需求（一个需求一份 state.md） | 按任务（一个任务一个文件） |
| **使用场景** | 智能体交接时读取上下文 | 大总管分配任务、智能体回报状态 |

### 两者如何协同

```
一个需求（REQ-001）
    │
    ├── Story File（story/state.md）
    │   └── 关注：这个需求现在在哪个阶段？上下文是什么？要传给谁？
    │
    └── Hub-Spoke Task Management（.tasks/）
        └── 关注：这个需求有哪些任务？谁在做？状态如何？
```

### 示例

**Story File 记录**：
```markdown
# REQ-001 状态
- 当前阶段: requirement-resolution
- 上下文: 用户想要双人对战功能
- 关键决策: 使用 WebSocket 而非 HTTP（DEC-xxx）
```

**Hub-Spoke Task Management 记录**：
```markdown
# task-001
- 需求ID: REQ-001
- 智能体: requirement-resolution
- 状态: running
- 产出文件: story/REQ-001-blueprint.md
```

### 关联方式

两者通过**需求 ID**（REQ-xxx）关联：
- Story File 记录需求的完整生命周期和上下文
- Task Management 记录需求下每个任务的分配和状态
- 两者的 ID 一致（REQ-xxx），方便交叉引用

## 📐 任务层级结构

```
需求（REQ-001）
  ├── 任务 1 → 需求澄清智能体（进行中）
  ├── 任务 2 → 需求理解智能体（待处理）
  ├── 任务 3 → 解决智能体（待处理）
  ├── 任务 4 → 验收智能体（待处理）
  └── 任务 5 → 需求澄清智能体（另一个相关需求，并发）
```

**关键设计**：
- **一个需求，多个智能体协同**
- **一个智能体，对同一个需求可能有多个任务**
- **每个任务必须关联需求 ID**

## 📁 文件结构规范（方案 A — 已确认）

### 任务文件结构

**✅ 已确认：状态放文件内容，不放在文件夹名**

```
workspace/
  └── .tasks/
        └── {agent-id}/
              └── REQ-{需求ID}/
                    ├── meta.md          # 需求元数据
                    ├── task-001.md      # 任务文件（状态在内容里）
                    └── task-002.md      # 任务文件（状态在内容里）
```

**对比（方案对比）**：

| 方案 | 结构 | 状态管理 | 复杂度 |
|------|------|---------|--------|
| ~~旧方案~~ | `.tasks/{agent}/REQ-xxx/{pending\|running\|completed}/` | 文件夹名携带状态，变更需移动文件 | 高（路径变化、引用复杂） |
| **方案 A（已确认）** | `.tasks/{agent}/REQ-xxx/task-xxx.md` | 状态在文件内容字段，变更只改内容 | 低（路径稳定） |

### 任务文件格式

```markdown
# 任务

- **需求ID**: REQ-001
- **任务ID**: task-001
- **智能体**: requirement-resolution
- **创建时间**: 2026-03-25 09:30
- **状态**: running
- **任务描述**: 需求澄清
- **产出文件**: story/REQ-001-blueprint.md
- **完成时间**: —
- **备注**: xxx
```

**状态可选值**：`pending` | `running` | `completed` | `failed` | `cancelled`

### 全局索引文件

```
workspace/.tasks/index.md
```

内容：
```markdown
# 任务总览

## REQ-001（用户双人对战功能）
| 任务ID | 智能体 | 状态 | 产出 |
|--------|--------|------|------|
| task-001 | requirement-resolution | ✅ 完成 | story/REQ-001-blueprint.md |
| task-002 | requirement-understanding | 🔄 进行中 | — |
| task-003 | solving | ⏳ 待处理 | — |

## REQ-002（xxx功能）
| 任务ID | 智能体 | 状态 | 产出 |
|--------|--------|------|------|
| task-004 | requirement-resolution | ⏳ 待处理 | — |
```

## 📋 协同格式约定

| 规范项 | 内容 |
|--------|------|
| **需求ID命名** | `REQ-{全局自增序号}`，如 `REQ-001` |
| **任务ID命名** | `task-{全局自增序号}`，如 `task-001` |
| **文件命名** | `story/{需求ID}-{阶段}.md`<br>例：`story/REQ-001-blueprint.md` |
| **阶段标记** | 每份文件头部注明 `阶段: requirement-resolution` 等 |
| **完成报告格式** | `✅ 完成: {任务}\n📄 产出: {文件路径}\n📝 说明: {简要描述}` |
| **共享目录** | 统一放在 `workspace/project/*/story/` 下 |
| **文件保留** | 中间产物至少保留到任务交付后 7 天 |

## 🔄 完整工作流程推演（方案 A）

### 第一步：大总管接收需求

```
用户 → 大总管：「我想要双人对战功能」
```

**大总管的操作**：
1. 分配需求 ID：`REQ-001`
2. 创建需求目录：`.tasks/requirement-resolution/REQ-001/`
3. 创建需求元数据：`.tasks/requirement-resolution/REQ-001/meta.md`
4. 创建任务文件：`.tasks/requirement-resolution/REQ-001/task-001.md`（状态：pending）
5. 更新 index.md（task-001 状态为 ⏳ 待处理）
6. 发送任务给需求澄清智能体（via `sessions_send`）

### 第二步：需求澄清智能体处理

**收到任务后**：
1. 创建任务文件：`.tasks/requirement-resolution/REQ-001/task-001.md`
2. 更新任务文件（状态：`pending` → `running`）
3. 更新 index.md（task-001 状态为 🔄 进行中）
4. Spawn subagent 执行需求澄清
5. Subagent 产出 `story/REQ-001-blueprint.md`
6. 更新任务文件：
   - 状态：`running` → `completed`
   - 产出文件：`story/REQ-001-blueprint.md`
   - 完成时间：`2026-03-25 09:35`
7. 更新 index.md（task-001 状态为 ✅ 完成）
8. 返回结果给大总管：
   ```
   「✅ task-001 完成
    📄 产出：story/REQ-001-blueprint.md
    📝 说明：已拆分为 6 个子需求，2 个技术方案」
   ```

### 第三步：大总管流转到需求理解智能体

**大总管的操作**：
1. 创建任务目录：`.tasks/requirement-understanding/REQ-001/`
2. 创建任务文件：`.tasks/requirement-understanding/REQ-001/task-002.md`（状态：pending）
3. 更新 index.md（创建 task-002，状态为 ⏳ 待处理）
4. 发送任务给需求理解智能体：
   ```
   「需求澄清产出：story/REQ-001-blueprint.md
    需求ID：REQ-001
    任务ID：task-002」
   ```

### 第四步：继续流转...

```
大总管 → 解决智能体（task-003）
大总管 → 验收智能体（task-004）
...
```

## 🔀 变更对比

### 变更前（V3.14.0）

- 大总管任务指派方式不统一
- 智能体感知任务依赖主会话对话
- 任务状态不透明，无全局视图
- 任务与需求关联缺失

### 变更后（V3.15.0）

- 大总管统一通过 `sessions_send` 分发任务
- 智能体通过任务文件管理本地任务状态
- 全局 index.md 提供透明的任务总览
- 每个任务必须关联需求 ID，支持同需求多任务
- 任务文件结构简化（方案 A），状态在文件内容而非文件夹名
