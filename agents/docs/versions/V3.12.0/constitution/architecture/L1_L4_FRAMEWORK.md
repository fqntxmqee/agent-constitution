# AI 时代的软件需求与资产标准化框架（L1-L4）

> **版本**: V1.0  
> **生效日期**: 2026-03-15  
> **状态**: 建设中

---

## 一、背景与目标

本框架旨在解决"一句话需求"或"多轮对话"到标准化系统资产的转化问题。通过 L1-L4 四层架构，结合 OpenSpec 规范，实现：

1. **消除歧义**：业务语言 → 技术语言的无损转换
2. **资产复用**：新需求优先通过"搭积木"复用现有功能点
3. **文档即代码**：利用 AI 反向生成，保证需求与实现的一致性

---

## 二、四层架构定义

### L1：领域层 (Domain)

**定义**：业务边界。代表一个完整的业务子系统或 Bounded Context，拥有独立的数据模型和业务规则。

**示例**：
- 交易域 (Trade)
- 商品域 (Product)
- 库存域 (Inventory)
- 用户域 (User)
- 营销域 (Marketing)

**AI 作用**：领域划分助手，分析代码库模块依赖，自动识别和校验领域边界是否清晰。

---

### L2：场景层 (Scenario)

**定义**：业务价值流。在特定领域下，用户为了达成某个完整目标所经历的一系列交互集合。

**示例**：
- 交易下单 (Place Order)
- 订单管理 (Order Management)
- 购物车结算 (Cart Checkout)
- 售后退款 (After-sales Refund)

**AI 作用**：场景聚类，将分散的用户故事（User Stories）自动归类到标准场景下，识别缺失的场景覆盖。

---

### L3：业务活动层 (Business Activity)

**定义**：行为原子。场景中的具体步骤，是对用户行为的抽象归纳，具有明确的输入输出状态变更。

**示例**：
- 下单渲染 (Order Rendering)
- 下单提交 (Order Submission)
- 库存预占 (Stock Reserve)
- 支付回调处理 (Payment Callback)

**AI 作用**：流程挖掘，通过日志或代码调用链，自动还原业务活动流程图，识别冗余或缺失的活动节点。

---

### L4：功能点层 (Function Point)

**定义**：服务能力。跨领域复用的最小技术单元，对应具体的逻辑计算或数据操作，是动态编排的"积木"。

**示例**：
- 商品查询 (get_product_detail)
- 库存校验 (check_stock_status)
- 优惠券计算 (calculate_coupon)
- 地址解析 (parse_address)

**AI 作用**：代码逆向工程，阅读代码，自动提取函数逻辑，抽象为标准功能点，并建立索引。

---

## 三、核心机制：动态编排与双向映射

### 3.1 正向路径：需求落地 (Business → Code)

```
1. 定位场景 (L2)：明确需求属于哪个场景
      ↓
2. 拆解活动 (L3)：将需求拆解为具体的业务活动
      ↓
3. 编排功能点 (L4)：
   - 复用：从功能点库中检索现有能力
   - 扩展/新增：定义新功能点
   - 生成编排 DSL：规定功能点的执行顺序、参数传递
      ↓
4. 代码生成/组装：AI 根据编排 DSL 自动组装代码
```

### 3.2 反向路径：资产治理 (Code → Business)

```
1. 代码扫描：AI Agent 定期扫描领域下的代码仓库
      ↓
2. 语义提取：
   - 识别独立的 Service 方法或 API 接口
   - LLM 分析业务语义
   - 归属判断：属于哪个 L3 业务活动？
      ↓
3. 功能点注册 (L4)：
   - 已存在 → 更新元数据
   - 新能力 → 自动创建，标记"待审核"
      ↓
4. 图谱更新：自动更新 L2→L3→L4 依赖图谱
```

## 四、与 OpenSpec 的映射

| OpenSpec 阶段 | L1-L4 映射 | 产出 |
|---------------|-----------|------|
| proposal | L1 领域 + L2 场景 | 需求草案（含四层定位） |
| specs | L3 业务活动 | 需求规格说明书 |
| design | L4 功能点 + 编排 | 技术设计文档（含 DSL） |
| tasks | 编排执行 | 任务清单 + 代码生成 |

## 五、功能点向量库（建设中）

### 目的

防止 AI 每次都"重新发明轮子"，通过 RAG 检索现有功能点。

### 实现方案

1. **向量化**：将所有 L4 功能点的语义描述存入向量数据库
2. **检索**：新需求进来时，先检索相似度 > 0.85 的现有功能点
3. **决策**：
   - 能复用 → 直接编排
   - 需扩展 → 扩展功能点定义
   - 需新增 → 新建功能点 + 编排

---

## 六、实施策略

### 阶段一：构建功能点知识库

- 输入：现有 Git 代码库
- AI 任务：遍历 Service 层方法，生成 L4 功能点清单
- 产出：{ID, 名称, 业务描述, 代码路径, 输入输出 Schema}

### 阶段二：建立活动与编排映射

- 输入：历史需求文档 + 代码调用链
- AI 任务：将 L4 功能点聚合成常见的执行序列
- 产出：L3 业务活动库及标准编排模板

### 阶段三：需求驱动的自动化闭环

- 输入：自然语言需求
- AI 任务：解析 → 匹配 → 缺口分析 → 生成
- 产出：标准化编排配置文件 + 骨架代码
