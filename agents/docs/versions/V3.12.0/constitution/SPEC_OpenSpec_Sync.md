## 一、目的与核心原则

### 1.1 双重保活原则（V3.12.0 新增）

**所有需要用户确认的文档，必须同时满足以下两个条件，流程方可往下走：**

1. ✅ **本地有**：文档文件存在于 `project/{项目名}/changes/{需求名}/` 目录
2. ✅ **飞书有**：文档已同步到飞书，且**飞书链接已发送给用户**

> ⚠️ **只有飞书链接已发送给用户并被用户查阅后，才能进入下一阶段。**
> 本地文件仅作为开发参考，飞书文档作为唯一真相源。

### 1.2 飞书作为唯一真相源

飞书文档作为**唯一真相源**，所有用户-facing 的规约内容以飞书为准。

本地文件作为**开发参考**，可由工具自动生成或手动维护。

## 二、核心原则

### 2.1 强制同步范围（V3.12.0 修订）

以下文档**必须**同步到飞书并发送给用户：

| 文档类型 | 同步时机 | 发送要求 | 说明 |
|----------|---------|---------|------|
| **澄清提案**（proposal.md） | 澄清完成后 | ✅ 链接发送给用户 | 用户确认后进入理解阶段 |
| **需求规格**（specs/requirements.md） | 理解完成后 | ✅ 链接发送给用户 | 用户确认蓝图后进入解决阶段 |
| **技术设计**（design.md） | 理解完成后 | ✅ 链接发送给用户 | 与 specs 同阶段发送 |
| **任务清单**（tasks.md） | 理解完成后 | ✅ 链接发送给用户 | 与 specs 同阶段发送 |
| **验收报告**（acceptance report） | 验收完成后 | ✅ 链接发送给用户 | 用户确认后进入交付阶段 |
| **交付报告**（delivery report） | 交付完成后 | ✅ 链接发送给用户 | 告知用户项目完成 |

### 2.2 规约按项目+需求隔离

每个**独立需求**有独立的规约目录：

```
project/{项目名}/changes/{需求名}/
├── proposal.md              # 需求提案
├── specs/
│   └── requirements.md     # 详细需求（L2/L3/L4）
├── design.md                # 技术设计
├── tasks.md               # 任务清单
├── acceptance_criteria.md  # 验收标准（V3.12.0 新增）
├── test-report.md          # 验收报告（可选）
├── story/                  # Story File 上下文
│   ├── state.md            # 当前故事状态
│   ├── context/            # 各阶段上下文
│   ├── decisions/          # 决策索引
│   └── feedback/           # 用户反馈
└── feishu-doc-urls.txt     # 飞书文档链接索引
```

## 三、用户确认节点与链接发送（V3.12.0 核心新增）

### 3.1 标准流程中的用户确认节点

```
澄清完成 → 发送 proposal 链接 → 用户确认 → 理解阶段
    ↓
理解完成 → 发送 specs + design + tasks 链接 → 用户确认 → 解决阶段
    ↓
验收完成 → 发送验收报告链接 → 用户确认 → 交付阶段
    ↓
交付完成 → 发送交付报告链接 → 告知用户
```

### 3.2 链接发送规范（V3.12.0 新增）

**发送格式**（主会话回复示例）：
```markdown
## 📄 规约已同步，请查阅

您的确认是进入下一阶段的前提：

| 文档 | 飞书链接 | 本地路径 |
|------|----------|----------|
| 澄清提案 | [查看](https://feishu.cn/docx/xxx) | proposal.md |
| 需求规格 | [查看](https://feishu.cn/docx/yyy) | specs/requirements.md |
| 技术设计 | [查看](https://feishu.cn/docx/zzz) | design.md |
| 任务清单 | [查看](https://feishu.cn/docx/aaa) | tasks.md |

请确认后告诉我「可以继续」，我即进入下一阶段。
```

### 3.3 主会话行为约束（V3.12.0 新增）

| 约束 | 说明 |
|------|------|
| ❌ 禁止跳过链接发送 | 未发送飞书链接给用户，禁止进入下一阶段 |
| ❌ 禁止本地替代 | 本地文件存在但飞书未同步，禁止进入下一阶段 |
| ❌ 禁止仅口头确认 | 必须有实际的飞书链接，用户确认「可以」 |
| ✅ 每次发送 | 每个用户确认节点，必须列出所有文档及其飞书链接 |
| ✅ 等待明确回复 | 等待用户明确说「可以」或「确认」，不自动推进 |

## 四、同步操作规范

### 4.1 创建飞书文档

```markdown
## 操作
1. 本地创建规约文件
2. 调用 feishu_doc create（title + content）
3. 获取返回的 doc_token
4. 将链接追加到 feishu-doc-urls.txt
5. 在主会话中发送链接给用户
```

### 4.2 更新飞书文档

```markdown
## 操作
1. 本地修改规约文件
2. 调用 feishu_doc write（content 覆盖）
3. 更新 feishu-doc-urls.txt 的最后更新时间
4. 在主会话中重新发送更新后的链接给用户

## 注意
- 使用 write 而非 append（append 可能导致重复）
- 更新后验证 block_count > 0
```

### 4.3 读取飞书文档

```markdown
## 操作
1. 调用 feishu_doc read 验证内容
2. 若 block_count == 0，立即 feishu_doc write 补充
3. 与本地文件交叉验证

## 验证清单
- [ ] block_count > 0
- [ ] 内容与预期一致
- [ ] 格式正确（表格/代码/标题）
```

## 五、冲突处理

### 5.1 本地与飞书不一致

**处理规则**：
1. 飞书内容 > 本地内容（飞书是真相源）
2. 若本地有而飞书没有，以飞书为准，本地覆盖
3. 若飞书有而本地没有，提示用户确认

### 5.2 飞书文档为空

**处理规则**：
1. 调用 feishu_doc read 检查 block_count
2. 若 block_count == 0，立即 feishu_doc write 补充
3. 记录错误到审计报告

## 六、审计检查

在**需求理解**和**需求交付**阶段，审计智能体应检查：

### 6.1 需求理解阶段
- [ ] 本地 proposal.md 存在且内容完整
- [ ] 飞书文档已创建，链接在 feishu-doc-urls.txt
- [ ] **飞书链接已在主会话中发送给用户**（V3.12.0 新增）
- [ ] 设计方案已同步到飞书

### 6.2 需求交付阶段
- [ ] 验收报告已同步到飞书
- [ ] **飞书链接已在主会话中发送给用户**（V3.12.0 新增）
- [ ] 飞书链接已回写到本地
- [ ] 本地文件与飞书内容一致（可抽样验证）

---

## 七、违规处理（V3.12.0 修订）

| 级别 | 情况 | 处理 |
|------|------|------|
| 🔴 严重 | 未发送飞书链接给用户即进入下一阶段 | 立即停止，补充链接后重新推进 |
| 🔴 严重 | 交付阶段飞书文档为空 | 立即补充，拒绝交付 |
| 🟡 一般 | 同步延迟 > 24h | 提醒同步，纳入下次审计 |
| 🟢 轻微 | 链接未回写 | 建议改进 |


### 8.3 同步检查清单

```markdown
## 同步前
- [ ] 本地文件已保存
- [ ] 飞书 doc_token 已获取

## 同步后
- [ ] feishu_doc read 验证 block_count > 0
- [ ] feishu-doc-urls.txt 已更新
- [ ] **飞书链接已在主会话中发送给用户**（V3.12.0 新增）

## 交付前
- [ ] 所有飞书文档 block_count > 0
- [ ] 链接全部回写本地
- [ ] **每份文档的飞书链接均已发送给用户**（V3.12.0 新增）
- [ ] 审计通过
```
