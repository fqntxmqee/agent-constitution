# 决策记录 — DEC-036

**决策编号**: DEC-036

**关联提案**: PROP-V3.14.0  
**决策日期**: 2026-03-25  
**决策者**: 用户（待确认）

## 1. 决策背景

### 1.1 当前状态
- 宪法规范 V3.13.0 定义了 8 个子智能体 + 1 个银河导航员（8+1 体系）
- 调试专家（debugger）已有完整的 AGENTS.md 工作规范（`agents/constitution/debugger/AGENTS.md`）
- 但调试专家**未被纳入宪法主流程**，处于"孤儿智能体"状态
- 在 CONSTITUTION.md 的智能体配置表中无调试专家条目
- 在 TEAM_ROLES.md 的角色映射表中无调试专家条目

### 1.2 触发事件
Rokid Glasses 项目联调过程中暴露出严重的调试流程空白：
- 蓝牙搜不到、连接卡住
- 服务器连不上、超时
- 应用 crash、后台断连
- 语音交互失败

在整个联调过程中，调试专家从未被调用，导致：
- 问题定位效率低下
- 修复缺乏根因分析（治标不治本）
- 系统缺少可调试性设计（无日志规范、无错误码、无 health check）

---

## 2. 决策内容

### 2.1 核心决策
将调试专家从"孤儿智能体"升级为正式子智能体，使体系从 8+1 扩展为 9+1。

### 2.2 具体措施

#### 措施 1：智能体体系扩展
- 调试专家正式成为第 9 个子智能体
- 定位为**辅助型智能体**（不占据主流程节点，而是在特定阶段被触发协作）

#### 措施 2：工作流触发点 A — 需求理解阶段
- 触发条件：跨系统集成、硬件交互、A/S 级复杂度、用户明确要求
- 职责：可调试性设计审查（日志埋点、分段测试、连通性检查、错误码、可观测性）
- 协作对象：需求理解智能体（脑洞整理师 💡）

#### 措施 3：工作流触发点 B — 需求解决阶段
- 触发条件：同一 bug 调试 ≥3 轮、跨服务/跨进程问题、根因不明的间歇性问题、用户要求
- 职责：根因分析（四阶段调试流程）
- 协作对象：需求解决智能体（功能魔法师 🪄）
- 铁律：NO FIXES WITHOUT ROOT CAUSE

---

## 3. 决策理由

| # | 理由 | 权重 |
|---|------|------|
| 1 | Rokid Glasses 联调暴露调试流程空白，问题定位效率极低 | 高 |
| 2 | 调试专家已有成熟的 AGENTS.md，纳入成本低 | 高 |
| 3 | 可调试性设计前移可大幅降低后期调试成本（预防优于治疗） | 高 |
| 4 | "NO FIXES WITHOUT ROOT CAUSE"原则可提升修复质量 | 中 |
| 5 | 消除"孤儿智能体"状态，使体系更完整 | 中 |

---

## 4. 备选方案分析

### 方案 A：维持现状（不纳入）
- **优点**：无需变更文件
- **缺点**：调试流程空白持续；类似 Rokid 联调问题将重复出现
- **结论**：❌ 不推荐

### 方案 B：仅在需求解决阶段纳入（单触发点）
- **优点**：变更范围更小
- **缺点**：缺少设计阶段的可调试性前移，仍是"事后补救"
- **结论**：⚠️ 次优方案

### 方案 C：在需求理解 + 需求解决两阶段纳入（双触发点）← 本提案采纳
- **优点**：前移防线 + 根因驱动，完整覆盖
- **缺点**：变更文件较多（9 个）
- **结论**：✅ 推荐方案
