## 变更概述

将**调试专家（debugger）🔬** 从当前的"孤儿智能体"状态正式纳入宪法主流程，使智能体体系从 **8+1**（8 个子智能体 + 1 个银河导航员）扩展为 **9+1**（9 个子智能体 + 1 个银河导航员）。同时在标准工作流中明确调试专家的两个触发点，建立与上下游智能体的正式协作关系。

---

## 变更原因

### 1. 问题背景：Rokid Glasses 项目联调教训

在 Rokid Glasses 项目联调过程中，遇到了大量涉及跨系统集成的 bug：

| 问题类别 | 具体表现 | 影响 |
|---------|---------|------|
| 蓝牙连接 | 设备搜不到、连接卡住 | 基本功能不可用 |
| 服务器通信 | 连不上、超时 | 核心流程中断 |
| 稳定性 | crash、后台断连 | 用户体验严重受损 |
| 语音交互 | 交互失败 | 核心功能不可用 |

**关键发现：** 尽管调试专家（debugger）已有完整的 AGENTS.md 定义，但由于未纳入宪法主流程，**在整个联调过程中从未被调用**。

### 2. 根因分析

| 根因 | 详情 |
|------|------|
| **流程缺位** | 调试专家不在标准工作流中，无明确触发点 |
| **缺乏可调试性设计** | 需求理解阶段未考虑日志、错误码、分段测试等 |
| **修复缺乏根因分析** | 功能魔法师独立修 bug，容易"治标不治本" |

### 3. 期望改进

- **前移防线**：在需求理解阶段就植入可调试性设计，将调试成本前移
- **根因驱动**：在需求解决阶段引入系统化根因分析，避免"盲修"
- **流程保障**：通过宪法规范确保调试专家在关键节点被正确触发

### 2. 工作流触发点

调试专家在标准工作流中有**两个明确触发点**：

#### 触发点 A：需求理解阶段 — 可调试性设计审查

```
需求澄清 → 需求理解（脑洞整理师）
                │
                ├─→ [触发调试专家] 可调试性设计审查
                │     ├── 日志埋点规范审查
                │     ├── 分段测试接口设计
                │     ├── 连通性检查设计
                │     ├── 错误码规范审查
                │     └── 可观测性设计审查
                │
                └─→ 产出《执行蓝图》（含可调试性设计章节）
```

**触发条件：**
- 需求涉及**跨系统集成**（蓝牙、网络、多端通信）
- 需求涉及**硬件交互**（设备连接、传感器、外设）
- 需求复杂度为 **A 级或 S 级**
- 用户明确要求可调试性设计

**产出：** 在 `design.md` 中新增「可调试性设计」章节，包含：
- 日志埋点方案（关键路径 + 状态变更 + 错误分支）
- 分段测试接口（各模块可独立验证）
- 连通性检查清单（端到端 health check）
- 错误码规范（统一编码 + 分类 + 描述）
- 可观测性指标（延迟、成功率、重试次数等）

#### 触发点 B：需求解决阶段 — 问题排查辅助

```
需求解决（功能魔法师）
    │
    ├─ 正常执行 → 继续
    │
    └─ 遇到 bug ──→ [触发调试专家] 根因分析
                       ├── Phase 1: 收集症状
                       ├── Phase 2: 模式匹配
                       ├── Phase 3: 形成假设
                       └── Phase 4: 验证后修复建议
                              │
                              └─→ 返回功能魔法师执行修复
```

**触发条件：**
- 同一 bug 调试超过 **3 轮**未解决
- 涉及**跨服务 / 跨进程**问题
- **根因不明**的间歇性问题
- 用户明确要求 "帮我 debug 这个 issue"

**铁律：NO FIXES WITHOUT ROOT CAUSE** — 在根因未明确之前，禁止修复代码。

### 3. 协作关系

#### 3.1 与需求理解智能体（脑洞整理师 💡）的协作

| 维度 | 说明 |
|------|------|
| **关系类型** | 辅助协作（调试专家为脑洞整理师提供可调试性审查） |
| **数据流** | 脑洞整理师发送 `design.md` 草稿 → 调试专家审查 → 返回审查意见 + 可调试性设计补充 |
| **产出归属** | `design.md` 的可调试性章节由调试专家编写，脑洞整理师整合入最终蓝图 |
| **冲突解决** | 若调试专家与脑洞整理师意见冲突，由银河导航员仲裁 |

#### 3.2 与需求解决智能体（功能魔法师 🪄）的协作

| 维度 | 说明 |
|------|------|
| **关系类型** | 辅助协作（调试专家为功能魔法师提供根因分析） |
| **数据流** | 功能魔法师发送 bug 描述 + 错误信息 → 调试专家分析 → 返回《调试报告》 |
| **产出归属** | 《调试报告》由调试专家产出，修复方案由功能魔法师执行 |
| **职责边界** | 调试专家**只做分析不做修复**，功能魔法师**依据根因执行修复** |

#### 3.3 与其他智能体的关系

| 智能体 | 关系 |
|--------|------|
| 需求澄清 🎯 | 无直接协作关系 |
| 需求验收 🔍 | 验收阶段可引用调试报告作为根因分析证据 |
| 需求交付 📦 | 无直接协作关系 |
| 进展跟进 ⏰ | 监控调试专家任务状态（调试耗时纳入进度统计） |
| 审计 🛡️ | 审计调试流程合规性（是否遵循四阶段流程、是否有根因才修复） |
| 总结反思 📝 | 复盘中分析调试专家的效果与改进点 |

## 影响分析

### 正面影响

| 影响点 | 说明 |
|--------|------|
| **问题定位效率** | 系统化根因分析 + 模式匹配，预计将平均调试时间缩短 50%+ |
| **修复质量** | "NO FIXES WITHOUT ROOT CAUSE"原则确保治本而非治标 |
| **系统可调试性** | 设计阶段前移，新系统天生具备日志、错误码、health check |
| **知识沉淀** | 调试报告形成可复用的调试模式库 |
| **团队完整性** | 消除"孤儿智能体"，体系更完整 |

### 潜在风险

| 风险 | 缓解措施 |
|------|---------|
| 触发过于频繁导致效率降低 | 明确触发条件阈值（3 轮未解决、跨系统等） |
| 与功能魔法师职责边界模糊 | 铁律："只分析不修复" + 审计监督 |
| 可调试性设计增加需求理解时长 | 仅 A/S 级强制、B 级推荐、C 级可选 |

### 向后兼容性

- ✅ 现有 8 个智能体的工作流不受影响
- ✅ 已有的需求文档格式无需变更
- ✅ 调试专家仅作为辅助角色加入，不改变主流程节点
- ⚠️ `design.md` 模板将新增「可调试性设计」可选章节

## 决策记录

**决策编号**: DEC-036  
**决策标题**: 将调试专家纳入宪法正式流程（8+1 → 9+1）  
**决策日期**: 2026-03-25  
**决策状态**: ⏳ 待批准

**决策内容**:
1. 调试专家从"孤儿智能体"升级为正式子智能体（第 9 个）
2. 在需求理解阶段增加可调试性设计审查触发点
3. 在需求解决阶段增加问题排查辅助触发点
4. 调试专家定位为"辅助型智能体"，不改变主流程节点顺序

**决策理由**:
- Rokid Glasses 联调暴露出调试流程空白
- 调试专家已有成熟的 AGENTS.md 定义，纳入流程成本低
- 可调试性设计前移可大幅降低后期调试成本

**影响范围**: 9 个文件需更新（详见更新文件清单）

详细决策记录见：`agents/docs/versions/V3.13.0/constitution/upgrade-to-V3.14.0/DECISION_LOG.md`
