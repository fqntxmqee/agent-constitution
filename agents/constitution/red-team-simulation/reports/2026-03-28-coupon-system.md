## 📋 需求概述

| 功能 | 描述 |
|------|------|
| 新用户注册 | 赠送 100 元券包 |

| 券类型 | 满减、折扣、包邮券 |

## 🏗️ 技术设计

| 组件 | 方案 |
|------|------|
| 架构 | 前端 (小程序/H5) → API 网关 (Nginx+Kong) → 优惠券服务 (Node.js) → Redis + MySQL |

| 超卖防护 | 数据库乐观锁 (version 字段) |
| 限流 | 单用户每秒最多领 3 张券 |
| 缓存策略 | 模板 24 小时过期，用户券 1 小时过期 |

- 不支持优惠券转赠
- 过期券不提醒
- 退款后优惠券不退回

# 🔴 红队挑战（按风险等级排序）

## 高风险挑战

### 1. 并发超卖风险 - Redis 与 DB 数据不一致

| 属性 | 描述 |
|------|------|
| **维度** | 风险识别 |
| **风险等级** | 🔴 高 |
| **影响范围** | 所有领券用户，可能导致超卖或少卖 |

**问题描述**: 
设计采用"Redis DECR + Lua 脚本"做库存扣减，同时用"数据库乐观锁"防超卖。但两者之间存在数据不一致风险：
- Redis 扣减成功后，DB 乐观锁失败（version 冲突），此时 Redis 库存已减但 DB 未写入
- 高并发下可能出现 Redis 显示有库存，但实际 DB 无法写入的情况
- Redis 宕机或主从切换时，库存数据可能丢失或不一致

**核心质疑**: Redis 和 MySQL 之间的库存数据如何保持最终一致性？当两者冲突时以哪个为准？

---

### 2. 最优优惠组合计算的性能问题

| 属性 | 描述 |
|------|------|
| **维度** | 性能扩展 |
| **风险等级** | 🔴 高 |
| **影响范围** | 所有结算用户，影响下单转化率 |

**问题描述**: 
"自动计算最优优惠组合（最多 3 张券）"在订单结算时实时计算。假设用户有 N 张可用券，组合数为 C(N,1) + C(N,2) + C(N,3)，当 N=50 时约 2 万种组合。
- 每次结算都要遍历计算，响应时间可能超过 SLA
- 大促期间并发结算请求激增，可能成为性能瓶颈
- 优惠券规则复杂（满减、折扣、包邮、互斥）时计算更复杂

**核心质疑**: 当用户有 50+ 张券时，最优组合计算能在 100ms 内完成吗？有没有预计算或缓存策略？

---

### 3. 缓存过期策略导致的数据一致性问题

| 属性 | 描述 |
|------|------|
| **维度** | 边界条件 |
| **风险等级** | 🔴 高 |
| **影响范围** | 所有查看优惠券的用户 |

**问题描述**: 
"模板 24 小时过期，用户券 1 小时过期"的缓存策略存在严重问题：
- 运营修改优惠券模板（如调整面值），24 小时内用户看到的仍是旧数据
- 用户券 1 小时过期后，如果用户正在浏览优惠券列表，会突然消失
- 缓存过期瞬间大量请求打到 DB，可能引发缓存击穿

**核心质疑**: 为什么模板缓存要 24 小时这么长？运营修改配置后如何及时生效？

## 中风险挑战

### 4. 限流策略过于简单，无法防羊毛党

| 属性 | 描述 |
|------|------|
| **维度** | 风险识别 |
| **风险等级** | 🟡 中 |
| **影响范围** | 营销预算安全，真实用户体验 |

**问题描述**: 
"单用户每秒最多领 3 张券"的限流策略存在漏洞：
- 羊毛党可以使用多个账号、多个设备、多个 IP 绕过限制
- 没有限制单用户每日/每月领取总数
- 没有设备指纹、行为分析等反作弊机制
- 新用户注册送 100 元券包，可能被批量注册薅羊毛

**核心质疑**: 如何识别和防御批量注册、自动化脚本领取？有没有风控系统集成？

### 5. 退款后优惠券不退回的业务逻辑漏洞

| 属性 | 描述 |
|------|------|
| **维度** | 假设挑战 |
| **风险等级** | 🟡 中 |
| **影响范围** | 退款用户，平台声誉 |

**问题描述**: 
"退款后优惠券不退回"的设定看似合理，但存在边界情况：
- 商家责任导致的退款（如发错货、商品质量问题），用户损失了优惠券
- 部分退款场景：订单含 3 件商品，退 1 件，优惠券是否按比例退回？
- 用户可能故意利用此规则：先用券下单，退款后券不退回，相当于"用过的券还能用"

**核心质疑**: 所有退款场景都不退券吗？是否应该区分退款原因（用户原因 vs 平台原因）？

**核心质疑**: 为什么不做过期提醒？是技术限制还是产品决策？有没有考虑过推送通知或短信提醒？

| 属性 | 描述 |

# 🔵 蓝队应对建议

## P0 优先级行动项

### P0-1: Redis+DB 库存一致性补偿机制

| 属性 | 描述 |
|------|------|
| **对应挑战** | 挑战 1 - 并发超卖风险 |
| **策略** | 方案设计 + 风险缓解 |
| **工作量** | 3-5 人天 |
| **优先级** | P0 |

**建议方案**: 采用"Redis 预扣减 + DB 最终确认 + 补偿机制"的三段式方案

**实施步骤**:
1. 领取请求到达时，先在 Redis 中用 Lua 脚本预扣减库存（原子操作）
2. 预扣减成功后，异步写入 DB，使用乐观锁
3. DB 写入失败时，将失败记录写入补偿队列
4. 定时任务扫描补偿队列，回滚 Redis 库存或重试 DB 写入
5. 设置 Redis 库存与 DB 库存的定期对账机制（如每小时）

**预期效果**: 确保 Redis 与 DB 库存最终一致，超卖风险降低 95%+

**验证方法**: 混沌工程测试 - 模拟 DB 写入失败场景，验证补偿机制是否生效
