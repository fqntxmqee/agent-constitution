## 🔴 红队挑战

### 挑战点 1: Redis 库存扣减与数据库不一致风险

| 属性 | 内容 |
|------|------|
| **维度** | 风险识别 |
| **风险等级** | 🔴 high |
| **影响范围** | 所有优惠券领取场景 |

**问题描述**:
设计使用 Redis DECR 扣减库存，但如果 Redis 扣减成功后数据库更新失败（网络故障/DB 异常），会导致库存数据不一致。

**核心质疑**:
如何保证 Redis 和 MySQL 的库存数据最终一致性？

**影响**:
超卖风险：实际库存少于显示，用户下单失败引发投诉

---

### 挑战点 2: 优惠券有效期跨天处理

| 属性 | 内容 |
|------|------|
| **维度** | 边界条件 |
| **风险等级** | 🟡 medium |
| **影响范围** | 跨时区用户 |

**问题描述**:
设计中使用 DATETIME 存储有效期，但未说明时区处理。用户跨时区旅行或服务器时区变更时，可能导致券提前或延后过期。

**核心质疑**:
有效期是使用 UTC 还是本地时间？用户跨时区如何处理？

**影响**:
用户体验问题：券意外过期或该过期未过期

---

### 挑战点 3: 热点券秒杀场景性能瓶颈

| 属性 | 内容 |
|------|------|
| **维度** | 性能扩展 |
| **风险等级** | 🔴 high |
| **影响范围** | 限时抢购活动 |

**问题描述**:
限时抢购场景下，单模板可能面临 10 万 + QPS 领取请求。仅靠 Redis 单 key 锁可能成为瓶颈。

**核心质疑**:
是否考虑过分片库存？如何支撑 10 万 + 并发领取？

**影响**:
系统崩溃：领取接口超时，用户投诉

---

### 挑战点 4: Redis 故障降级方案缺失

| 属性 | 内容 |
|------|------|
| **维度** | 依赖分析 |
| **风险等级** | 🟡 medium |
| **影响范围** | Redis 故障场景 |

**问题描述**:
设计强依赖 Redis 缓存库存和用戶券，但未说明 Redis 不可用时的降级策略。

**核心质疑**:
Redis 宕机时是否可以降级到直接查库？如何保证性能？

**影响**:
服务中断：Redis 故障导致无法领券/用券

---

### 挑战点 5: "最优组合计算仅支持 3 张券"假设合理性

| 属性 | 内容 |
|------|------|
| **维度** | 假设挑战 |
| **风险等级** | 🟢 low |
| **影响范围** | 多券用户 |

**问题描述**:
设计限制最多 3 张券组合，但用户可能有 10+ 张可用券，组合数爆炸。

**核心质疑**:
为什么是 3 张而不是 5 张？算法复杂度考虑过吗？

**影响**:
用户损失：未能使用最优优惠，体验差

---

### 挑战点 6: 过期券无提醒机制

| 属性 | 内容 |
|------|------|
| **维度** | 用户体验 |
| **风险等级** | 🟡 medium |
| **影响范围** | 所有持券用户 |

**问题描述**:
已知限制中明确"过期券不提醒"，这会导致用户券浪费，引发不满。

**核心质疑**:
是否可以在过期前 1 天发送推送/短信提醒？

**影响**:
用户流失：券过期未使用，对平台产生负面印象

---

### 挑战点 7: 退款后优惠券不退回的业务风险

| 属性 | 内容 |
|------|------|
| **维度** | 风险识别 |
| **风险等级** | 🟡 medium |
| **影响范围** | 退款订单 |

**问题描述**:
用户退款后券不退回，如果是因为商家责任（发错货/质量问题），用户会认为不公平。

**核心质疑**:
是否应该区分退款原因？责任方不同处理方式是否应不同？

**影响**:
客诉风险：用户投诉平台规则不合理

|------|------|
